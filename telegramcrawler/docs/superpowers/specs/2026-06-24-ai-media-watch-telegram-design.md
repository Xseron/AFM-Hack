# AI Media Watch — Telegram Crawler & Risk Analyzer (MVP)

**Дата:** 2026-06-24
**Статус:** Дизайн утверждён, готов к планированию реализации

## Цель

Сервис, который получает на вход Telegram-каналы (ссылки/@username), краулит их
контент, анализирует на признаки незаконного игорного бизнеса, финансовых пирамид
и мошеннических схем, формирует объяснимую риск-оценку и отдаёт приоритизированный
список каналов для проверки через REST API.

Каналы для анализа поставляет внешнее приложение (оно парсит Instagram/TikTok и
извлекает оттуда ссылки на Telegram). Наша система отвечает **только** за краулинг
и анализ Telegram.

## Границы (scope)

**В MVP:**
- Приём списка каналов через REST API
- Краулинг последних N постов канала (текст, подписи, ссылки, картинки) через Telethon
- Гибридный анализ: лексикон/regex-префильтр (RU + KZ) → LLM-скоринг через OpenRouter
- Vision-анализ картинок (баннеры казино) через vision-модель OpenRouter
- Риск-оценка канала 0–100 с категориями, цитатами-доказательствами и объяснением
- Хранение результатов в БД
- REST API: постановка на анализ, отчёт по каналу, приоритизированный список

**Вне MVP (YAGNI, но архитектура оставляет место):**
- Видео/аудио-анализ (Whisper, извлечение кадров)
- Фоновый планировщик периодического мониторинга
- Веб-UI / дашборд
- Discovery новых каналов (граф связей, поиск)
- Очередь задач / асинхронные джобы

## Стек

| Компонент | Выбор | Причина |
|-----------|-------|---------|
| Язык | Python 3.11+ | Требуется Telethon |
| API | FastAPI | Быстрый REST, Pydantic-схемы |
| Краулинг | Telethon (MTProto, user-аккаунт) | Полный доступ к истории/медиа |
| LLM + Vision | OpenRouter (OpenAI-совместимый, `openai` SDK, `base_url=https://openrouter.ai/api/v1`) | Один ключ ко многим моделям, выбор модели через конфиг |
| Хранилище | SQLite + SQLAlchemy | Ноль настройки; миграция на Postgres тривиальна |
| Схемы | Pydantic | Валидация запросов/ответов и structured output |

Модели задаются в конфиге: дешёвая текстовая модель для скоринга + vision-модель
для картинок. Смена модели — без правок кода.

## Архитектура

Модули с единственной ответственностью:

```
api/        FastAPI: эндпоинты приёма каналов и выдачи отчётов
crawler/    Telethon-клиент: загрузка постов + медиа по @username
prefilter/  Лексикон (RU+KZ) + regex: дешёвый отсев и пометка кандидатов
analyzer/   OpenRouter: per-post риск-оценка, категории, evidence, vision-OCR картинок
scoring/    Агрегация post-оценок → риск-оценка канала + приоритет
storage/    SQLAlchemy-модели и репозиторий
config/     Ключи (API_ID/HASH, OPENROUTER_API_KEY), модели, пороги
```

Каждый модуль тестируется изолированно: crawler и OpenRouter мокируются, реальной
сети в тестах нет.

## Поток данных

1. `POST /channels` — принимает список ссылок/@username (синхронный ответ)
2. **crawler** тянет последние N постов (по умолчанию ~50): текст, подписи, ссылки,
   скачивает картинки
3. **prefilter** прогоняет двуязычный лексикон (казино, ставки, 1xbet/мостбет,
   «гарантированный доход», «реферал», «×2 за день» и т.д.) → помечает подозрительные
   посты, считает дешёвую эвристику
4. **analyzer** отправляет помеченные посты (текст + картинки через vision) в OpenRouter
   со structured output → по каждому посту: категории нарушений, уверенность,
   цитаты-доказательства, краткое объяснение
5. **scoring** агрегирует в риск-оценку канала (0–100), список сработавших категорий,
   приоритет
6. **storage** сохраняет канал, посты, оценки
7. API отдаёт отчёт и приоритизированный список

## Категории риска

Под формулировку задачи:

- `illegal_gambling` — реклама нелицензированных онлайн-казино
- `financial_pyramid` — финансовые пирамиды
- `guaranteed_income` — обещания гарантированного дохода
- `aggressive_investment` — агрессивные инвестиционные призывы
- `referral_scheme` — реферальные схемы
- `hidden_engagement` — скрытые способы вовлечения аудитории

## Риск-оценка и объяснимость

- Риск канала: 0–100, взвешенная агрегация post-оценок (бо́льший вес у явных
  категорий gambling/pyramid).
- Каждый отчёт содержит **объяснение** и **цитаты-доказательства** из конкретных
  постов (требование объяснимости из ТЗ).
- Приоритизация: список каналов сортируется по убыванию риск-оценки.

## REST API

| Метод | Путь | Описание |
|-------|------|----------|
| POST | `/channels` | Поставить канал(ы) на анализ; синхронно возвращает отчёт(ы) |
| GET | `/channels/{username}` | Риск-отчёт: score, категории, доказательства, объяснение |
| GET | `/channels?sort=risk` | Приоритизированный список для проверки |
| GET | `/health` | Проверка живости |

## Модель данных (SQLite)

- **channel**: id, username, title, status (`ok`/`error`/`pending`), error_reason,
  risk_score, last_analyzed_at
- **post**: id, channel_id, tg_message_id, date, text, has_media, media_paths
- **post_assessment**: id, post_id, categories (JSON), confidence, evidence_quotes
  (JSON), explanation, model_used
- **channel_assessment** (или поля в channel): агрегированный score, категории,
  объяснение

Схема рассчитана на будущий мониторинг: `last_analyzed_at` и история постов
позволяют добавить периодический воркер без переделок.

## Обработка ошибок

- Приватный / несуществующий канал → `status=error`, `error_reason`, не валит батч
- Флуд-лимиты Telegram → retry с экспоненциальным backoff
- Таймауты/ошибки OpenRouter → пометка поста, продолжение анализа остальных
- Любая ошибка по одному каналу не прерывает обработку остальных в батче

## Тестирование (TDD)

- **prefilter**: юнит-тесты на лексикон RU+KZ (срабатывания и отсутствие ложных)
- **scoring**: агрегация post-оценок → корректный канал-score и приоритет
- **analyzer**: фейковые ответы OpenRouter (моки), проверка парсинга structured output
- **crawler**: мок Telethon-клиента
- Реальной сети в тестах нет

## Конфигурация (env)

- `TELEGRAM_API_ID`, `TELEGRAM_API_HASH`, `TELEGRAM_SESSION` — доступ Telethon
- `OPENROUTER_API_KEY` — доступ к LLM/vision
- `OPENROUTER_TEXT_MODEL`, `OPENROUTER_VISION_MODEL` — выбор моделей
- `POSTS_PER_CHANNEL` (по умолчанию 50), пороги риск-оценки
